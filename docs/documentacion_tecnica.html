1. Arquitectura del pipeline

Flujo interno de pipeline.py

El pipeline transforma los datos crudos en los archivos analíticos que consume el tablero. No re-entrena el modelo: el modelo RF fue entrenado en la entrega 2 y se carga serializado desde models/rf_model.joblib.

Entrada
CSVs fuente
  • Directorio data/raw/
  • Datos de eventos FullStory, tenants y técnicos
ETL — Fase 1
etl.run_phase1()
  • Limpieza de tenants, técnicos y eventos
  • Unificación de fuentes y normalizacion de IDs
Feature Engineering — Fase 2
etl.build_feature_store()
  • 57 variables por técnico en dos bloques: comportamental (A) y contexto del tenant
  • Ventana de observación: primeros 14 días del técnico en FMA
Scoring — Fase 3
RF pipeline (joblib)
  • predict_proba: score de retención [0,1] por técnico
  • Output: analytical_dataset.parquet
Scoring: < 0.1 seg (muestra)
Interpretabilidad — Fase 4
SHAP TreeExplainer
  • Top-3 factores explicativos por técnico
  • Output: shap_values.parquet y feature_importance.parquet
SHAP: ~1.5 seg (muestra)
Agregación — Fase 5
Agregación por tenant
  • Score medio, cuartil y métricas de retención real por cliente
  • Output: tenant_scores.parquet
Total pipeline: ~3.8 seg (muestra n=140) · estimado <10 min en producción (n=66K técnicos).

5. Preprocesamiento

Pipeline scikit-learn del modelo RF

Pipeline([
    ("pre", ColumnTransformer([
        ("num", "passthrough", top_num), # 15 variables numéricas
        ("cat", OneHotEncoder(handle_unknown="ignore"), top_cat), # 5 categóricas → OHE
    ])),
    ("clf", RandomForestClassifier(
        class_weight="balanced",
        random_state=42,
        # hiperparámetros optimizados via RandomizedSearchCV con TimeSeriesSplit
    ))
])
  • Desbalance de clases: class_weight='balanced' — proporción 3:1 en datos de entrenamiento.
  • Valores nulos: conteos se imputan como 0 cuando no hubo actividad; categóricas como "Unknown".
  • Validación: TimeSeriesSplit para respetar la naturaleza temporal de los datos.

7. Re-entrenamiento

Proceso de actualización del modelo (R21 / R16)

El modelo se re-entrena manualmente cuando hay nuevos datos disponibles. Frecuencia recomendada: bisemanal, o cuando lleguen datos de ≥50 técnicos nuevos.

1
Actualizar los datos fuente
Copiar los nuevos CSVs al directorio data/raw/.
2
Ejecutar el módulo de entrenamiento
source .venv/bin/activate
python -m src.run
3
Copiar el nuevo modelo
cp entrega_2/code/outputs/models/RF.joblib entrega_3/models/rf_model.joblib
4
Actualizar metadatos del modelo
Actualizar entrega_3/models/rf_model_metadata.json con las métricas y features de la nueva corrida (ver entrega_2/code/outputs/reports/run_metadata.json).
5
Regenerar los archivos analíticos
python pipeline.py
6
Verificar métricas
Confirmar que las métricas del nuevo modelo cumplen los umbrales R5-R9 antes de publicar.
Criterios de alerta para re-entrenamiento forzado
  • AUC en validación cae por debajo de 0.70 (R5).
  • La distribución de app_version_max en nuevos datos difiere significativamente (nueva versión de app lanzada).
  • Se incorporan >500 técnicos nuevos al dataset (suficiente volumen para re-entrenar).
8. Despliegue

Streamlit Community Cloud (R17)

1
Subir el código a GitHub
Subir el directorio entrega_3/ a un repositorio GitHub. Solo el código fuente, sin CSVs.
2
Conectar en Streamlit Cloud
En streamlit.io/cloud, conectar el repositorio y apuntar a app.py.
3
Configurar los datos analíticos
Subir los Parquet generados como archivos de datos del app, o conectar a un bucket S3/GCS si se dispone de infraestructura.
El tablero queda accesible vía navegador para ≥3 usuarios sin instalación local (R17). Acceso activo en: fma-adoption-intelligence-4czccfxlxzkfdm6prmcns8.streamlit.app
